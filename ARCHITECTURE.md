# Chanakya — Engine Reference

A UCI chess engine in C++. This document is a map of what's implemented so an AI
assistant can reason about the engine without needing the full source pasted in.

> **Rule for the reader (AI):** Do not assume implementation details that aren't
> stated here (e.g. exact constant values, exact pruning conditions, exact bit
> layouts). If something matters for the task and isn't spelled out below,
> **ask for the specific file/function** instead of guessing. Wrong assumptions
> about this codebase (e.g. "it uses PVS" or "it has tapered eval") will lead to
> bad patches.

---

## 1. High-Level Architecture

```
main.cpp        → newGame(), loadPolyBook(), UCI()
core/uci        → UCI loop, position/go/setoption handling, time management
core/board      → mailbox board (120-sq), game state, FEN, Zobrist key trigger
core/bitboard   → 64-bit bitboards, magic-bitboard attack tables, masks
core/move       → move encode/decode, makeMove/takeMove, null move
core/movegen    → pseudo-legal move generation (all moves / captures only)
core/zobristKeys→ Zobrist hash tables (piece/castle/side)
core/polyglot   → Polyglot .bin opening book reader
core/utils      → attack queries (isUnderAttack), draw detection, perft glue
eval/evaluation → static evaluation function
search/search   → iterative deepening, alpha-beta, quiescence
search/transpositionTable → TT storage, packing, replacement policy
```

Two coordinate systems are used simultaneously:
- **120-square mailbox** (`Board::pieces[120]`), 10 files × 12 ranks with an
  off-board border (`offBoard` sentinel) so sliding-piece edge detection during
  *attack-mask generation* doesn't need bounds checks. Squares here are called
  "sq120".
- **64-square bitboard index** (0–63, `PIECE_TYPE`/rank-major), used for all
  `U64` bitboards. Squares here are called "sq64".

Conversion tables `sq120To64[120]` and `sq64To120[64]` translate between them,
built once in `initSquareMappings()`.

> Two parallel piece representations are kept in sync at all times:
> `Board::pieces[120]` (mailbox, source of truth for "what piece is on this
> square") and `Bitboard::pieces[13]` (one U64 per piece type, source of truth
> for "which squares does this piece occupy"). Every mutation
> (`addPiece`/`removePiece`/`movePiece` in `move.cpp`) updates **both**. If you
> change move-making logic, both must stay consistent or bitboard-based
> attack/mobility code will silently desync from the mailbox.

## 2. Board State (`Board` class)

Holds: `pieces[120]` mailbox, `side`, `castlePermission` (4-bit flags:
CASTLE_WK=8, CASTLE_WQ=4, CASTLE_BK=2, CASTLE_BQ=1), `fiftyMove`, `enPassantSq`,
`checkSq` (square of the king in check, or `SQ_NONE`), `ply`, `pieceCount[13]`,
`material[2]` (incremental material sums), `positionKey` (Zobrist hash), and a
`history[1024]` ring of `MoveInfo` for unmake.

FEN parsing (`parseFen`) rebuilds bitboards via `bitboard->initBoard(this)`,
determines `checkSq` for the side to move, and computes the initial Zobrist key
from scratch (`generatePositionKey`). All hash updates *after* that are
incremental (XOR in/out) — see §9 below.

## 3. Bitboards & Magic Bitboards (`Bitboard` class)

- One `U64 pieces[13]` per piece type (index 0 = empty/unused, 1–6 = white
  P/R/N/B/Q/K, 7–12 = black P/R/N/B/Q/K — see `PIECE_*` enum in `defs.hpp`).
- Non-sliding attack tables, precomputed once per square: `pawnAttacks[2][64]`,
  `kingAttacks[64]`, `knightAttacks[64]`.
- Sliding pieces (rook/bishop) use **magic bitboards**:
  - `rookAttacks[64]` / `bishopAttacks[64]` = the *unblocked* attack mask per
    square (edge bits stripped depending on rank/file, per magic-bitboard
    convention).
  - `rookBlockers[64]` / `bishopBlockers[64]` = every possible blocker subset
    for that mask, generated by `generateBlockerBoards` (2^popcount
    combinations).
  - `rookMagics[64]` / `bishopMagics[64]` are **precomputed constant** magic
    numbers (hardcoded arrays at the bottom of `bitboard.cpp`), found offline
    by `findMagic()` (brute-force search, not run at engine startup).
  - `rookLookupTable[64][4096]` / `bishopLookupTable[64][4096]` are filled at
    startup: for each blocker subset, `index = (blocker * magic) >> (64-12)`,
    store the legal-move bitboard for that blocker configuration.
  - Runtime queries: `getRookAttacks(sq)` / `getBishopAttacks(sq)` in
    `utils.cpp` — mask the full occupancy against the attack mask, index into
    the lookup table with the magic multiply-shift.
  - **12-bit index everywhere** (`>> (64-12)`, table size 4096) — this is a
    fixed shift, not per-square relevant-bit-count shifting like some magic
    implementations use. Table is intentionally larger than the strict
    per-square minimum.
- Other precomputed masks: `fileMasks[8]`, `rankMasks[8]`, `isolatedPawnMask[64]`
  (adjacent files), `passedPawnMask[2][64]` (files ±1, ranks ahead, per side).
- `getPieces(side)` ORs the 6 piece bitboards for a side (or all 12 for BOTH).

> If you need exact magic-number values, blocker-generation order, or the
> precise edge-masking logic in `init_rookAttacks`/`init_bishopAttacks`, ask —
> don't assume a "standard" magic bitboard layout, this one has some
> engine-specific choices (fixed 12-bit shift, mask-stripping done inline
> during attack generation rather than via a separate "relevant bits" table).

## 4. Move Encoding (`move.hpp` / `move.cpp`)

A move is a single packed `int`:

```
bits 0-6    from square (sq120, 7 bits)
bits 7-13   to square (sq120, 7 bits)
bits 14-17  captured piece (4 bits)
bits 18-21  promoted piece (4 bits)
bit 22      en-passant flag   (0x400000)
bit 23      castle flag       (0x800000)
bit 24      pawn-start flag   (0x1000000)
```

`buildMove/moveFrom/moveTo/moveCapturePiece/movePromotionPiece` pack/unpack
these. `CAPTURE_FLAG` (0x3c000) and `PROMOTION_FLAG` (0x3c0000) are bitmasks
used to test "is this a capture / promotion" without decoding the piece.

`makeMove(move)`:
1. Pushes undo info (`MoveInfo`) to history.
2. Handles en-passant capture / castling rook relocation specially.
3. Updates Zobrist hash incrementally for castle rights / en-passant / side.
4. Updates `fiftyMove` counter, sets new en-passant square on double pawn push.
5. Applies the move to both the mailbox and the bitboards (`movePiece`,
   `addPiece`, `removePiece` — all keep both representations in sync, see §1).
6. Handles promotion (remove pawn, add promoted piece).
7. Flips side to move, then **checks if the moving side's own king is now
   attacked** — if so, calls `takeMove()` to undo and returns `false`
   (this is how legality is enforced — see §5, moves are generated
   pseudo-legally and filtered here).
8. Sets `checkSq` for the new side to move if the enemy king is now attacked.

`makeNullMove`/`takeNullMove` exist for null-move pruning (flips side without
moving a piece; refuses to null-move while in check — enforced by the caller
in search, see §7).

## 5. Move Generation (`movegen.cpp`)

**Pseudo-legal generation, filtered for legality at `makeMove()` time** (no
separate "is this move legal" pre-check beyond that). Two entry points:

- `generateMoves()` — all pseudo-legal moves (quiet + captures + castling +
  en passant + promotions).
- `generateCaptureMoves()` — captures only (used by quiescence search).

Both return `std::vector<std::pair<int,int>>` — **(move, score)** pairs. The
score is assigned at generation time by `addCaptureMove`/`addQuiteMove`/
`addEnPassantMove` (see §6 — move ordering is computed *during* generation,
not as a separate pass).

- Pawns: handled with explicit per-side loops (white shifts +10/+9/+11 in
  sq120 space, black −10/−9/−11), promotion unrolled into 4 moves
  (Q/R/B/N) per promoting push/capture, en passant checked against
  `board->enPassantSq`.
- Knights/King (`genNonSlidingMoves`): iterate set bits of the piece bitboard,
  AND the precomputed attack table with `~friendlyPieces`, split into
  capture/quiet based on enemy-occupancy test.
- Rooks/Bishops/Queens (`genSlidingMoves`): same pattern but the attack
  bitboard comes from `getRookAttacks`/`getBishopAttacks` (magic lookup,
  occupancy-dependent) rather than a static table.
- Castling: explicit square-emptiness + `isUnderAttack` checks on the transit
  square (not full path — see note below) for each of the 4 castle rights,
  gated on `board->checkSq == SQ_NONE`.

> Castling legality only calls `isUnderAttack` on **one intermediate square**
> per side (e.g. F1 for white kingside), relying on `checkSq == SQ_NONE` to
> cover "king currently in check" and the empty-square checks to cover "path
> blocked". Whether this is actually sufficient for O-O-O's B-file square, or
> whether there's an edge case, is worth verifying by reading the exact code
> rather than assuming — ask if this matters for what you're doing.

`isGameOver()` (utils.cpp) checks for no-legal-moves by generating pseudo-legal
moves and testing `makeMove`/`takeMove` on each until one succeeds.

## 6. Move Ordering

Ordering scores are assigned **inline during move generation**, then
`swapWithBest()` does a per-node partial selection sort (pick highest-scoring
remaining move each iteration — *not* a full `std::sort` up front, and *not*
a lazily-sorted list; it's O(n²) worst case per node but avoids sorting moves
that get pruned early).

Priority (highest score wins):
1. **TT / PV move** — if the transposition-table move for this position is in
   the move list, its score is overwritten to `2,000,000` (see `alphaBeta`,
   not in movegen itself).
2. **Captures** — MVV-LVA: `MvvLva[victim][attacker] + 1,000,000`
   (`MvvLva` table in `movegen.cpp`, victim-major).
3. **En passant** — scored via `MvvLva[PAWN][PAWN] + 1,000,000` (same tier as
   equal-value captures).
4. **Killer moves** — 2 killer slots per ply (`searchController->killers[ply][0/1]`),
   scored `900,000` / `800,000`.
5. **History heuristic** — `history[piece][toSquare]`, incremented by
   `depth * depth` whenever a quiet move raises alpha (see §7). If a quiet
   move has no history yet, it gets a small deterministic tiebreak
   (`1 + (fromSquare % 1000)`) instead of a flat 0, so early move-ordering
   isn't a total coin-flip.

There is **no separate SEE (static exchange evaluation)** — capture ordering
is MVV-LVA only, and quiescence delta pruning (§7) uses a flat material
margin instead of SEE.

## 7. Search (`search/search.cpp`)

**Algorithm shape:** iterative deepening → fail-hard alpha-beta with:
null-move pruning, check extension, late move reduction (LMR), quiescence
search at the leaves, and a transposition table. This is **plain alpha-beta,
not PVS** (no null-window search on non-first moves aside from the LMR
re-search path below) and there are **no aspiration windows** (each iterative
deepening depth searches the full `[-AB_BOUND, AB_BOUND]` window).

**`searchPosition()`** — entry point:
- If `useBook` is on, tries `getRandBookMove()` (Polyglot book) first and
  returns immediately if found.
- Otherwise iterative deepening loop `depth = 1..searchController->depth`,
  calling `alphaBeta(-AB_BOUND, AB_BOUND, depth, true)` each iteration.
- After each completed depth, prints a UCI `info` line (score/depth/nodes/
  time/pv), reconstructing the PV via `transpositionTable->getLine(depth)`
  (replays TT best-moves move by move, not a separate PV table/triangular
  array).
- Mate scores are reported as `mate N` when `bestScore` crosses
  `AB_BOUND - MAX_DEPTH` (see mate-scoring note below).

**`alphaBeta(alpha, beta, depth, doNull)`:**
- `depth <= 0` → drops into `quiescence()`.
- Time check every 2048 nodes (`checkTimeUp`).
- Draw detection: `isRepetition()` (scans `board->history` back to the last
  fifty-move reset) or `fiftyMove >= 100` → return 0 (skipped at the root,
  `ply == 0`).
- Depth cutoff at `MAX_DEPTH` (64) → static eval fallback.
- **Check extension:** `depth++` if `inCheck`.
- **TT probe:** if stored depth ≥ current depth, use the stored bound
  (alpha/beta/exact flag) to cut off or return immediately; TT move is
  extracted for move ordering regardless of depth sufficiency.
- **Null move pruning:** conditions — not in check, `ply != 0`, `depth >= 4`,
  `bigPieceCount(side) > 1` (i.e. not just king+pawns, avoids zugzwang risk).
  Reduction is fixed **R = 4** (`depth - 4`, not depth-dependent/adaptive).
  Verification: none (no re-search at reduced depth to confirm a null-move
  beta cutoff).
- **Main move loop:** move-ordering score bump for the TT move, then
  `swapWithBest` per iteration (see §6).
  - **LMR:** applies when `depth >= 3 && i >= 4` (move index) and the move is
    quiet, not in check, not a promotion, and not a killer at this ply.
    Reduction: 1 (moves 4-7), 2 (moves 8-15), 3 (moves 16+). If the reduced
    search beats alpha, **re-searches at full depth** (standard LMR
    re-search).
  - On a quiet move that raises alpha, history table is bumped by
    `depth * depth`.
  - Beta cutoff: stores killer move (if quiet), updates fail-high stats
    (`fh`/`fhf`, used only for the `ordering` % printed in `info`), stores TT
    entry with `BetaFlag`, returns beta immediately (fail-hard).
  - No legal moves found: checkmate (`-AB_BOUND + ply`) if in check, else
    stalemate (0).
  - TT store at the end with `ExactFlag` (alpha improved) or `AlphaFlag`
    (alpha never improved — fail-low).

**`quiescence(alpha, beta, checkPly)`:**
- Same time/draw/depth guards as `alphaBeta`.
- **Stand-pat**: static eval used as a lower bound unless the position is "in
  check and still within `MAX_QCHECK_PLY` (8) of check-extension budget" —
  in which case it fully expands **all** moves (not just captures) instead of
  stand-patting, to avoid missing forced mates/escapes while in check.
  `checkPly` counts consecutive in-check plies inside qsearch and this
  expansion stops being used past depth 8 to bound the search.
- **Delta pruning:** if not in check and `eval + QUEEN_VALUE + 200 < alpha`,
  prune immediately (flat margin, not SEE-based).
- Otherwise generates captures only (`generateCaptureMoves`) — or all moves if
  `expandFull` — orders them with the same `swapWithBest`, recurses.
- No captures/moves and in the "expand full" (in-check) branch → mate score.

**Time management (`handleSearch` in `uci.cpp`):** supports `movetime` or
`wtime/btime/winc/binc` + `movestogo` (defaults to 30 if not given by GUI).
Per-move budget = `(time + inc) / movestogo`, capped at 70% of *raw* remaining
clock (guards against `movestogo == 1` allocating everything), minus a fixed
50ms move-overhead safety margin, floored at 1ms.

**Mate scoring:** `MATE = AB_BOUND - MAX_DEPTH`. Scores stored in the TT are
shifted by `ply` on store/retrieve so mate distances stay correct regardless
of which node stored them (standard mate-in-TT handling).

> Not implemented: aspiration windows, PVS/null-window search on non-PV
> moves, futility pruning, static-null-move (reverse futility) pruning, SEE,
> multi-threaded (Lazy SMP) search, counter-move heuristic, singular
> extensions. If a task asks to "improve search," these are the likely
> next-step candidates — but confirm current behavior against the actual
> function before assuming an addition is safe (e.g. null-move R=4 with no
> verification search is fairly aggressive; changing it interacts with the
> zugzwang guard `bigPieceCount > 1`).

## 8. Evaluation (`eval/evaluation.cpp`)

`evalPosition()` returns a score **from the side-to-move's perspective**
(negated at the end if `board->side == BLACK`). Components, roughly in the
order computed:

1. **Material** — `board->material[WHITE] - board->material[BLACK]`
   (incrementally maintained elsewhere, not recomputed here). Piece values
   (`defs.hpp`): P=208, N=781, B=825, R=1276, Q=2538, K=50000.
2. **Pawns** (per pawn, both colors, mirrored for black via `MIRROR_64`):
   - Piece-square table (`PawnTable`).
   - `PawnSupport` (+5) if defended by another pawn.
   - `PawnIsolated` (−10) if no friendly pawn on adjacent files
     (`isolatedPawnMask`).
   - `PawnPassed[rank]` (0,5,10,20,35,60,100,200 by rank-from-start) if no
     enemy pawn on same/adjacent files ahead (`passedPawnMask`).
   - Doubled-pawn penalty (`DoublePawnPenality = -5`, scaled by count per
     file) computed in a **separate loop over files** after the per-pawn
     loop, not folded into it.
3. **Knights** — PST (`KnightTable`) + mobility bonus (see below).
4. **Bishops** — PST (`BishopTable`) + mobility bonus. Bishop-pair bonus
   (`BishopPair = 40`) is added once at the end if `pieceCount[B] >= 2` for a
   side (does **not** check the bishops are on opposite colors).
5. **Rooks** — PST (`RookTable`) + mobility. Plus:
   - `rookSupportSameFile`/`rookSupportSameRank` (+10/+5) if it X-rays
     another friendly rook on the same file/rank via its own attack pattern.
   - Open-file bonus (`RookOpenFile = 20`) if no pawns of either color on
     that file, else semi-open (`RookSemiOpenFile = 10`) if no *friendly*
     pawns on that file.
   - **7th-rank bonus** (labeled "Taretary penetration" in a code comment —
     read as "territory"): if the rook is on rank 7 (white) / rank 2
     (black), +20 per enemy pawn it attacks on that rank, and +40 flat if
     it's within 1 file of the enemy king.
6. **Queens** — mobility bonus (rook+bishop attack union), open/semi-open
   file bonus (half the rook values), and a same-file support bonus if it
   X-rays a friendly rook.
7. **King safety** (`kingSafety(kingSq, color)`) — only active if the king is
   on one of 6 specific "castled" squares (B1/C1/G1 white, B8/C8/G8 black;
   central/uncastled kings get **no** shield/storm evaluation at all):
   - Pawn shield: penalty per missing shield pawn
     (`brokenPawnShieldPenalty = -20`), extra `exposedKingPenalty = -30` if
     that file has *no* friendly pawn anywhere (fully open in front of king).
   - Pawn storm: `pawnStormPenalty[color][rank]` — enemy pawns advancing
     toward the king on the shield files are penalized more the closer they
     get.
   - Minor-piece king-zone pressure: `kingZoneAttackPenalty = -10` per enemy
     knight/bishop attack that overlaps the king's own attack zone
     (`kingAttacks[kingSq]`).
8. **King PST** — `KingE` (endgame table, centralizing) or `KingO`
   (opening/middlegame table, staying back/castled) selected by a **binary**
   `isEndgame` flag, not interpolated/tapered between the two.

**Mobility bonus** (`MobilityBonus[phase][piece][mobilityCount]`) — indexed by
`isEndgame` (0 or 1) and popcount of (attacks & ~ownPieces). Precomputed
tables per piece type for rook/knight/bishop/queen; pawns and king have none.

**`isEndgame` determination** (binary switch, used for both king PST and
mobility tables — **not a tapered/interpolated eval**, one or the other
applies wholesale): true if *any* of —
- both sides' material ≤ `EndGame_Material`, OR
- combined material ≤ `2 × EndGame_Material`, OR
- both sides have ≤ 4 pawns.

`EndGame_Material = 1 rook + 2 knights + 2 pawns + king value` worth of
material (see constant definition in `evaluation.cpp`).

> There is no phase-interpolated ("tapered") evaluation anywhere in this
> engine — every phase-dependent term (king PST, mobility) is a hard
> either/or switch on `isEndgame`. If asked to add tapered eval, that's a
> structural change, not a tweak — flag that explicitly rather than quietly
> reinterpreting `isEndgame` as a 0-256 phase value.

## 9. Zobrist Hashing (`core/zobristKeys.hpp`, `core/utils.cpp`)

Precomputed random constant tables: `pieceKeys[13][120]`, `castleKeys[16]`
(indexed by the 4-bit castle-rights value), `sideKey`. `Board::positionKey` is
built from scratch once (`generatePositionKey`, on FEN parse) and then
maintained **incrementally**: `hashPiece`, `hashEnPassant`, `hashCastle`,
`hashSide` XOR the relevant key in/out on every state change inside
`makeMove`/`takeMove`/`addPiece`/`removePiece`/`movePiece`.

This is a **separate hash system** from the Polyglot book key (`getPolyKey()`
in `polyglot.cpp`, using its own `piecePolyKeys`/`castlePolyKeys`/
`enPassantPolyKeys`/`sidePolyKey` tables) — the two keys are not
interchangeable, don't mix them up.

## 10. Opening Book (`core/polyglot.cpp`)

Reads a standard Polyglot `.bin` file (`loadPolyBook`) into an
`unordered_map<U64, vector<polyEntry>>` keyed by Polyglot hash. Big-endian
byte order in the file is corrected via `endian_swap_u16/32/64`.
`getRandBookMove()` picks uniformly at random among all book moves for the
current position (no weighting by the book's `weight` field despite it being
parsed). `extractPolyMove` decodes Polyglot's compact move format and
special-cases the king-captures-rook castling notation (`e1h1`→`e1g1`, etc.)
into the engine's own from/to notation.

## 11. UCI Interface (`core/uci.cpp`)

Standard UCI commands (`uci`, `isready`, `setoption`, `ucinewgame`,
`position`, `go`, `stop`, `quit`) plus debug-only extensions: `d` (print
board), `move <uci>` (play a move then let the engine reply, 2s fixed think
time), `undo`, `book` (dump book entries for current position), `eval`
(print static eval), `cls`/`clear`.

Search runs on a dedicated `std::thread`; `stop`/`quit`/a new `go` all set
`searchController->stopped = true` and join the thread before proceeding.
UCI options exposed: `Hash` (spin, MB, resizes the TT — see §12) and
`usebook` (check).

## 12. Transposition Table (`search/transpositionTable.cpp`)

Single flat array, size = largest power of 2 that fits the requested MB
(`resize`). Each `TableData` slot: `smp_data` (packed score/depth/flag/move),
`smp_key` (= `positionKey XOR smp_data`, a lockless-SMP-style verification
trick — even though the engine is currently single-threaded for search), and
`age`.

**Packing** (`pack_tableData` / `extract_*` inline functions in the header):
score in low 16 bits (offset by `INFINITE`), depth in bits 16-21, flag in
bits 22-23, move in bits 24+.

**Replacement policy** (`add()`), first matching rule wins: empty slot →
replace; same position (verified via the XOR trick) → replace; deeper search
→ replace; slot aged out (`currentAge - entry.age > 2`) → replace; same depth
but newer search → replace. Otherwise the existing entry is kept.

**Indexing:** `positionKey & sizeMask` (no separate "index" bits vs "verification"
bits split — the full key is used for both the index and, via the XOR trick,
the verification).


> `perft.cpp` isn't covered elsewhere in this doc: `perft(depth)` recursively
> counts leaf nodes from pseudo-legal moves (skipping illegal ones via
> `makeMove`'s return value), and `perftTest(depth)` does the same per-root-move
> with timing, for move-generator correctness/speed testing — not used by
> search or eval.

## 14. Known Gaps (don't assume these exist)

- No SEE, no aspiration windows, no PVS, no futility/reverse-futility pruning,
  no singular extensions, no counter-move table, no tapered/phase-interpolated
  eval, no multi-threaded search.
- Bishop-pair bonus doesn't check opposite-color bishops.
- King safety terms only fire for kings on the 6 hardcoded "castled" squares.
- Null-move pruning has no verification search.
